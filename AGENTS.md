# AGENTS.md

## 项目边界

- 当前授权项目路径：本仓库根目录。
- 创建、修改、删除、移动等写操作只能发生在本仓库根目录及其子路径内。
- 旧项目目录已取消授权，不要读取或写入。
- Shell 环境：PowerShell。

## 安全要求

- API key 不得出现在对话、命令行参数、日志或仓库文件中。
- 本项目会安装到用户级目录 `$HOME/plugins/Codex-Llmcaller`，但除非用户明确要求安装或验证客户端，否则不要主动写用户目录。
- 不要回滚用户未要求回滚的修改。

## 防爆半径与权限沙盒

- 默认以本仓库根目录为唯一读写工作区；除非任务必须且用户明确授权，不读取旧项目目录、用户级敏感目录、系统目录或其他仓库。
- 所有创建、修改、删除、移动等写操作必须限制在本仓库根目录及其子路径内。涉及用户级插件目录、Codex 配置目录、云资源、数据库或其他外部系统时，必须先说明目的并获得用户明确要求。
- 任何 AI 输出都先视为草稿或建议。不要直接执行不可逆操作，例如删除数据、重置仓库、覆盖用户配置、应用云资源变更、批量删除文件或修改生产凭证。
- 禁止为了调试而读取、打印、提交或记录真实 API key、secret、token、cookie、生产连接串、私钥或其他凭证。文档和测试只能使用占位符、环境变量名或 mock 值。
- 不在 profile、headers、示例配置、日志、测试快照或开发记录中保存 secret-bearing 字段；如必须表达凭证来源，只写 `secretName`、环境变量名或 fingerprint。

## 确定性与外部依赖隔离

- 开发核心逻辑时优先使用 Mock Provider、固定响应和本地测试夹具，不直接依赖真实 LLM API。路由、规则引擎、能力判断、错误分类、输出模式和 fallback 逻辑必须可重复单测。
- 真实 provider 调用只用于安装验证、手动自测或用户明确要求的集成验证；不得把真实网络调用作为普通单元测试的前置条件。
- 核心模块必须能在外部网络、真实数据库、Docker Daemon 或云服务不可用时本地闭环运行。新增功能应提供离线测试路径和明确的降级行为。
- 任何依赖当前日期、模型列表、官方 API 能力或价格/配额的事实，开发前必须查官方文档或现有代码，不基于记忆写死。

## 智能体架构与关注点分离

- 对外保持统一入口和稳定工具契约。新增底层 provider、路由策略或专业化 Agent 时，不让前端/用户直接耦合内部实现。
- 路由层只做决策：选择 provider/profile/model、输入输出模式、是否联网、是否使用 fallback。路由层不读取 secret、不执行外部副作用、不生成业务结论。
- 专业化 Agent 或执行器应能力原子化：例如材料识别、文档提取、逻辑评审、图像生成、语音生成、工具调用执行分别有独立接口、schema 和测试。
- 暴露给用户、前端或 MCP tool 的输入输出必须结构化、可校验、向后兼容。新增字段应有默认值和迁移策略，避免破坏旧 profile 和旧脚本。
- Function calling、外部工具执行或本地脚本调用必须走 allowlist、参数校验和权限边界；默认只返回建议或结构化调用计划，不自动执行高风险动作。

## 数据抽象与脱敏过滤

- 面对文件、图片、音频、PDF、日志、表格等输入时，先做材料识别和归一化，再进入模型调用或路由；不要把逻辑硬编码到某个特殊模板、公司格式或单一文件结构上。
- 新增 RAG、知识库或样例数据时，必须使用无害化、泛化、脱敏后的内容。不得把真实客户数据、生产日志、内部邮件、私有凭证或个人信息注入 LLM 上下文。
- 测试夹具应最小化且可公开；如需要模拟敏感结构，使用人工构造数据，并确认 release-check 能拦截 `.data/`、`secrets.json`、`config.json`、备份文件和临时输出。
- 模型输入日志和错误信息必须做敏感信息裁剪或脱敏；不要为了排错把完整请求体、base64 媒体、API key、Authorization header 或本地隐私路径写入日志。

## 工作流程

- 修改前先读取相关文件，不要基于记忆直接改。
- 使用 `apply_patch` 做文件编辑。
- 处理任务时按“阅读现状 -> 明确小目标 -> 最小化修改 -> 验证 -> 回看”的节点循环推进。
- 禁止无必要的大重构。每个节点只做能被测试和审查的小增量；若确需重构，先说明边界、兼容策略和回滚风险。
- 每完成一个节点，回看检查节点产出是否符合节点预期和整体目标；未通过当前节点验证，不进入下一个开发节点。
- 每次代码改动必须配套相应自动化测试或说明为什么无法自动化验证。核心逻辑新增行为必须有单元测试覆盖。
- 每次涉及代码、脚本、配置、测试、文档或 skill 修改时，必须同步更新 `development-deployment-log.md`；提交前检查开发与部署记录是否覆盖本次 diff。
- 每次变更的最终说明必须包含：改了什么、验证了什么、是否需要重新运行 `setup.mjs` 和重启 Codex Desktop、遗留风险或未验证项。
- 修改后至少运行：

```powershell
node ./plugins/Codex-Llmcaller/scripts/server.test.mjs
node ./plugins/Codex-Llmcaller/scripts/self-test.mjs
node ./plugins/Codex-Llmcaller/scripts/release-check.mjs
```
